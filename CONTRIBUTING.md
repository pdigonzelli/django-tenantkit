# Contributing

Thanks for your interest in contributing! This project is maintained by a single author, so all changes go through Pull Requests.

## How to contribute

1. **Fork** this repository
2. **Create a branch** from `main` with a descriptive name:
   ```
   git checkout -b fix/some-bug
   git checkout -b feat/some-feature
   ```
3. **Make your changes** — keep them focused and minimal
4. **Test your changes** before submitting
5. **Open a Pull Request** against `main`

## Branch naming

| Prefix | Use for |
|--------|---------|
| `feat/` | New features |
| `fix/` | Bug fixes |
| `docs/` | Documentation only |
| `refactor/` | Code restructuring, no behavior change |
| `test/` | Tests only |
| `chore/` | Maintenance, dependencies |

## What makes a good PR

- One concern per PR — don't mix features with bug fixes
- Clear title and description (use the template)
- Include tests if you're adding or changing behavior
- Update docs if the public API changes
- Keep diffs small and readable

## What I won't merge

- Breaking changes without prior discussion (open an issue first)
- PRs that don't follow the branch naming or template
- Code without tests for new behavior
- Unrelated changes bundled together

## Reporting bugs or suggesting features

Open an [issue](../../issues) before writing code. This avoids wasted effort if the idea doesn't fit the project's direction.

## Code style

- Follow the existing conventions in the codebase
- Run the linter/formatter before pushing
